## 新增需求

### 需求: 基于关键词的论文检索
系统必须接受用户输入的关键词或研究主题，同时查询 arXiv 和 Semantic Scholar API，返回带元数据的论文列表。

#### 场景: 使用有效关键词搜索
- **当**用户提交一个非空的关键词时
- **则**系统查询 arXiv API 和 Semantic Scholar API 并返回聚合结果

#### 场景: 使用空关键词搜索
- **当**用户提交一个空关键词时
- **则**系统返回错误消息，提示用户输入关键词

### 需求: 论文元数据增强
系统必须为每篇论文补充引用数、影响力评分和开源代码可用性标识。

#### 场景: 论文有可用的引用数据
- **当**从 Semantic Scholar 获取论文时
- **则**系统在论文元数据中包含引用数和影响力评分（MVP 阶段 influence_score = citation_count）

#### 场景: 论文有开源代码
- **当**论文有已知的代码仓库（来自 Papers With Code 或 GitHub 链接）时
- **则**系统将 `hasCode` 标志设为 true，并存储仓库 URL

### 需求: 按项目维度去重与幂等存储
系统必须按 `project_id + arxiv_id` 维度存储论文，同一项目内相同 arxiv_id 的论文不重复插入。

#### 场景: 新论文保存到数据库
- **当**检索到一篇项目内不存在的新论文时
- **则**系统写入 paper 表，关联 project_id

#### 场景: 项目内重复论文跳过插入
- **当**同一关键词或不同关键词搜索到项目内已存在的相同 arxiv_id 论文时
- **则**系统跳过插入，不产生重复记录

#### 场景: 同一论文跨项目独立存储
- **当**同一篇论文出现在两个不同项目中时
- **则**系统在每个项目中分别创建独立的 paper 记录

### 需求: 结果聚合与排序
系统必须合并来自多个数据源的结果，按论文标题相似度（Levenshtein 距离 < 0.15）去重，并支持按相关度、引用数或发表日期排序。

#### 场景: 去重重叠结果
- **当**同一篇论文同时出现在 arXiv 和 Semantic Scholar 的结果中时
- **则**仅返回一条记录，包含合并后的元数据（优先使用 arXiv 的摘要和分类，补充 Semantic Scholar 的引用数和影响力评分）

#### 场景: 按引用数排序
- **当**用户请求 `sort=citation` 时
- **则**论文按 citation_count 降序返回

#### 场景: 按发表日期排序
- **当**用户请求 `sort=date` 时
- **则**论文按 publish_date 降序返回

#### 场景: 按相关度排序
- **当**用户请求 `sort=relevance` 或未指定 sort 参数时
- **则**论文按与搜索关键词的匹配度降序返回

### 需求: 搜索结果分页
系统必须对搜索结果进行分页，支持可配置的页码和页大小（默认 20，最大 100），并返回总记录数。

#### 场景: 获取第一页结果
- **当**用户搜索时未指定页码
- **则**系统返回前 20 条结果及总记录数

#### 场景: 获取后续页
- **当**用户请求第 2 页，页大小为 10 时
- **则**系统返回第 11-20 条结果

#### 场景: 页大小超过上限
- **当**用户请求的页大小超过 100 时
- **则**系统按最大 100 条返回

### 需求: 多源检索部分失败处理
当 arXiv 和 Semantic Scholar 中某一数据源请求失败时，系统必须返回已成功获取的数据，并在响应中标注哪些数据源失败。

#### 场景: arXiv 成功，Semantic Scholar 失败
- **当**arXiv API 返回有效数据但 Semantic Scholar API 超时或报错时
- **则**系统返回 arXiv 的数据，并在响应中标注 `sources: {arxiv: success, semantic_scholar: failed}`，前端展示黄色提示条

#### 场景: 两个数据源全部失败
- **当**arXiv 和 Semantic Scholar 均返回错误时
- **则**系统返回空列表，并返回错误码 EXTERNAL_API_FAILED，附带失败详情

#### 场景: 部分失败时异步任务状态
- **当**某一数据源失败但另一数据源成功时
- **则**异步任务状态标记为 SEARCHED（非 FAILED），但响应中包含失败数据源的警告信息

### 需求: 单次搜索数量上限
单次搜索每个外部数据源最多获取 100 条论文结果，超出部分不请求，返回前 100 条并按排序规则合并。

#### 场景: 外部 API 返回大量结果
- **当**arXiv API 返回超过 100 条结果时
- **则**系统只请求前 100 条（10 页 × 每页 10 条），忽略后续结果

#### 场景: 外部 API 返回结果不足 100 条
- **当**外部 API 返回的结果总数少于 100 条时
- **则**系统返回所有可用的结果

### 需求: arXiv 元数据提取
系统必须从 arXiv Atom XML 响应中提取以下字段：arxiv_id、标题、摘要、作者列表、发表日期、主分类、PDF 链接。

#### 场景: 完整元数据提取
- **当**arXiv API 返回有效 Atom 条目时
- **则**系统提取 title、summary、authors (JSON 数组)、published 日期、categories 中的第一个主分类、PDF 链接

#### 场景: 作者映射到 JSONB
- **当**arXiv 返回作者列表时
- **则**系统将作者姓名存入 JSONB 数组，格式 `["Author One", "Author Two"]`

#### 场景: 代码标记初始为假
- **当**新论文从 arXiv 检索到时
- **则**hasCode 初始为 false，code_url 为 null，等待后续 Semantic Scholar 数据补充

### 需求: 异步检索任务
系统必须将论文检索作为异步任务执行，返回唯一 taskId，前端通过 WebSocket 监听任务进度。

#### 场景: 触发异步检索
- **当**用户提交关键词并触发检索时
- **则**系统立即返回 `{ taskId, status: "SEARCHING" }`，检索在后台执行

#### 场景: 相同关键词的重复检索
- **当**用户对同一项目使用相同关键词再次检索时
- **则**系统检查 Redis 缓存和 DB 中是否已有该关键词的结果，若有则跳过外部 API 调用

#### 场景: 通过 WebSocket 推送进度
- **当**异步检索任务执行时
- **则**系统通过 STOMP 协议向 `/topic/task/{taskId}` 推送进度消息（stage: "SEARCHING"/"SEARCHED"/"FAILED"，progress: 0-100）

### 需求: 检索三层数据流
系统必须按 Redis 缓存 → 数据库 → 外部 API 的顺序检索论文，避免重复调用外部 API。

#### 场景: Redis 缓存命中
- **当**用户搜索关键词且 Redis 中存在该项目的缓存时
- **则**系统直接返回缓存结果，不调用数据库或外部 API

#### 场景: 数据库命中
- **当**Redis 缓存未命中但数据库中存在该项目已保存的检索结果时
- **则**系统返回数据库结果并写回 Redis 缓存（TTL 24 小时）

#### 场景: 新检索
- **当**Redis 和数据库均无结果时
- **则**系统调用 arXiv 和 Semantic Scholar API，写入数据库并写回 Redis 缓存

### 需求: 检索任务状态持久化
系统必须将检索任务状态持久化到 task_status 数据库表，应用重启或页面刷新后仍可查询。

#### 场景: 创建检索任务记录
- **当**用户触发检索时
- **则**系统在 task_status 表中创建一条记录（taskId, task_type=SEARCH, status=SEARCHING, progress=0）

#### 场景: 页面刷新后恢复任务状态
- **当**前端刷新页面后通过 taskId 查询任务状态时
- **则**系统从 task_status 表返回该任务的最新状态和进度

### 需求: 外部 API 限流保护
系统必须控制对 arXiv 和 Semantic Scholar API 的请求频率，避免触发限流。

#### 场景: arXiv API 请求频率控制
- **当**系统连续请求 arXiv API 时
- **则**每次请求之间至少间隔 3 秒

#### 场景: Semantic Scholar API 请求频率控制
- **当**系统连续请求 Semantic Scholar API 时
- **则**5 分钟内请求数不超过 100 次

#### 场景: 触发限流时指数退避
- **当**外部 API 返回限流错误时
- **则**系统使用指数退避策略（1s, 2s, 4s）重试，最多 3 次

### 需求: 检索任务状态查询
系统必须提供接口供前端查询检索任务的当前状态（SEARCHING / SEARCHED / FAILED）。

#### 场景: 查询进行中的检索任务
- **当**前端通过 taskId 查询检索任务状态时
- **则**系统返回当前阶段和进度百分比

#### 场景: 查询已完成的检索任务
- **当**前端通过 taskId 查询已完成的检索任务时
- **则**系统返回 SEARCHED 状态和论文总数

### 需求: 单篇论文查询
系统必须提供通过 paper_id 获取单篇论文详情的接口，用于论文详情抽屉和推荐论文查看原文。

#### 场景: 查询存在的论文
- **当**前端通过 paper_id 查询论文详情时
- **则**系统返回完整论文元数据 (标题/摘要/作者/日期/引用数/影响力/代码/PDF/分类)

#### 场景: 查询不存在的论文
- **当**前端通过 paper_id 查询不存在的论文时
- **则**系统返回 RESOURCE_NOT_FOUND 错误码
