## 新增需求

### 需求: 批量论文分析
系统必须采用多阶段深度分析策略：逐篇理解 → 交叉对比 → 创新生成，以确保分析质量而非追求速度。

#### 场景: 阶段 1 - 逐篇深度理解
- **当**用户对一组已检索论文触发分析时
- **则**系统并行对每篇论文发起独立的 LLM 调用，提取核心问题、方法论、关键贡献、局限性、关键词、具体方法/数据集

#### 场景: 阶段 2 - 交叉对比分析
- **当**所有论文的逐篇理解完成后
- **则**系统将 N 篇论文的结构化摘要汇总，发起一次 LLM 调用进行交叉分析：主题聚类、方法对比、结论冲突检测、共识点、方法论缺口、数据集缺口

#### 场景: 阶段 3 - 创新机会生成
- **当**交叉对比分析完成后
- **则**系统基于阶段 2 的交叉分析结果，发起一次 LLM 调用生成研究空白、base 论文推荐和创新点

#### 场景: 论文数量不足时的分析
- **当**可供分析的论文少于 5 篇时
- **则**系统返回警告，提示分析质量可能受限，但仍继续执行多阶段分析

### 需求: 论文全文提取
系统必须在 LLM 分析前获取论文全文（非仅摘要），使用 5 层回退链确保信息完整性。公式、图表对分析质量至关重要。

#### 场景: LaTeX 源提取成功
- **当**arXiv 提供 LaTeX 源码 (.tar.gz) 时
- **则**系统下载并解压，提取 .tex 文件中的纯文本，保留公式源码（$...$、\begin{equation} 等）

#### 场景: 视觉模型提取
- **当**LaTeX 源不可用或提取内容过短 (<500 字符) 时
- **则**系统将 PDF 逐页转为 PNG (300 DPI)，交由视觉模型 (qwen2.5-vl) 提取文本，保留图表和公式的视觉信息

#### 场景: ar5iv HTML 转换
- **当**LaTeX 源和视觉提取均失败时
- **则**系统尝试访问 ar5iv HTML 版本，提取文本（公式转为 MathJax，图表丢失但公式结构保留）

#### 场景: PDFBox 纯文本提取
- **当**ar5iv HTML 不可用时
- **则**系统使用 PDFBox 提取纯文本（公式乱码、图表丢失，但摘要和正文可提取）

#### 场景: 仅使用摘要回退
- **当**所有全文提取方式均失败时
- **则**系统使用论文摘要作为回退，并在分析结果中标注"该论文缺少全文"

#### 场景: 提取策略可配置
- **当**系统启动时
- **则**从 application.yml 读取全文提取策略的顺序列表，按配置顺序组装责任链

#### 场景: 使用默认策略顺序
- **当**用户未在配置文件中指定策略顺序时
- **则**系统使用默认顺序：LaTeX → 视觉模型 → ar5iv → PDFBox → 摘要回退

#### 场景: 自定义策略顺序
- **当**用户在配置文件中指定了策略顺序时
- **则**系统按配置顺序组装责任链，支持跳过某些策略或调整优先级

#### 场景: 提取策略标记
- **当**全文提取完成后
- **则**返回结果中包含策略名称、文本长度、警告信息，供后续分析质量评估

### 需求: 逐篇深度理解
系统必须对每篇论文独立调用 LLM，提取结构化的深度理解结果。

#### 场景: 提取论文核心要素
- **当**对单篇论文发起深度理解时
- **则**系统提取核心问题（1 句话）、方法论分类（理论/实验/综述/对比）、关键贡献（2-3 条）、局限性（2-3 条）、关键词（5-8 个）、具体方法/数据集

#### 场景: 逐篇分析并行执行
- **当**多阶段分析开始时
- **则**系统使用线程池并行发起 N 次 LLM 调用（N = 论文数量），每篇独立处理

#### 场景: 单篇分析失败
- **当**某篇论文的逐篇分析失败时
- **则**系统标记该论文为分析失败，跳过并继续处理其他论文，最终报告中标注缺失篇数

### 需求: 交叉对比分析
系统必须将所有论文的逐篇理解结果汇总，进行深度的交叉对比分析。

#### 场景: 研究主题聚类
- **当**交叉分析开始时
- **则**系统将论文分为 2-4 个研究主题簇，每个簇包含相关论文列表

#### 场景: 方法对比
- **当**交叉分析进行时
- **则**系统构建方法对比矩阵，标注每篇论文使用的具体方法和数据集

#### 场景: 结论冲突检测
- **当**交叉分析进行时
- **则**系统识别相互矛盾的结论，标注冲突双方的论文及具体矛盾点

#### 场景: 缺口识别
- **当**交叉分析进行时
- **则**系统识别方法论缺口（什么方法没人用）和数据集缺口（什么数据没人覆盖）

### 需求: 分析数量上限限制
系统单次分析最多处理 20 篇论文，超出部分截断并返回警告。

#### 场景: 论文数量超过上限
- **当**用户触发分析时项目中有超过 20 篇论文
- **则**系统只分析前 20 篇，并在响应中告知截断信息

#### 场景: 论文数量在合理范围
- **当**用户触发分析时论文数量在 5-20 篇之间
- **则**系统分析全部论文

### 需求: 异步分析任务
系统必须将创新分析作为异步任务执行，返回唯一 taskId，前端通过 WebSocket 监听任务进度。

#### 场景: 触发异步分析
- **当**用户触发创新分析时
- **则**系统立即返回 `{ taskId, status: "ANALYZING" }`，分析在后台执行

#### 场景: 分析任务完成
- **当**分析任务完成时
- **则**系统通过 WebSocket 向 `/topic/task/{taskId}` 推送 ANALYZED 状态和分析结果摘要

#### 场景: 通过 WebSocket 推送三阶段进度
- **当**异步分析任务执行时
- **则**系统通过 STOMP 协议推送阶段进度：
  - 阶段 1: "正在逐篇深度理解... ({completed}/{total})"，进度 0-85%
  - 阶段 2: "正在进行交叉对比分析..."，进度 85-90%
  - 阶段 3: "正在生成创新建议..."，进度 90-100%

### 需求: 分析任务状态持久化
系统必须将分析任务状态持久化到 task_status 数据库表，应用重启或页面刷新后仍可查询。

#### 场景: 创建分析任务记录
- **当**用户触发分析时
- **则**系统在 task_status 表中创建一条记录（taskId, task_type=ANALYSIS, status=ANALYZING, progress=0）

#### 场景: 页面刷新后恢复任务状态
- **当**前端刷新页面后通过 taskId 查询任务状态时
- **则**系统从 task_status 表返回该任务的最新状态和进度

### 需求: 结构化 LLM 输出
系统必须使用 JSON Schema 验证每个阶段的 LLM 分析响应，格式不符合要求时自动重试。

#### 场景: LLM 返回有效 JSON
- **当**LLM 响应符合预期 JSON Schema 时
- **则**系统解析为 Java 对象并传递给下一阶段

#### 场景: LLM 返回无效 JSON
- **当**LLM 响应无法解析为有效 JSON 或不符合 JSON Schema 时
- **则**系统记录错误并重新请求，最多重试 2 次

#### 场景: 阶段 1 JSON Schema (逐篇理解)
- **当**逐篇分析请求发出时
- **则**LLM 必须返回包含 coreQuestion、methodologyCategory、keyContributions[]、limitations[]、keywords[]、methodsAndDatasets 的 JSON 对象

#### 场景: 阶段 2 JSON Schema (交叉分析)
- **当**交叉分析请求发出时
- **则**LLM 必须返回包含 topicClusters[]、methodComparison[]、conflicts[]、consensusPoints[]、methodologyGaps[]、datasetGaps[] 的 JSON 对象

#### 场景: 阶段 3 JSON Schema (创新生成)
- **当**创新生成请求发出时
- **则**LLM 必须返回包含 gaps[] (研究空白)、basePapers[] (推荐论文)、innovationPts[] (创新点列表) 的 JSON 对象

### 需求: 分析任务状态查询
系统必须提供接口供前端查询分析任务的当前状态（ANALYZING / ANALYZED / FAILED）。

#### 场景: 查询进行中的分析任务
- **当**前端通过 taskId 查询分析任务状态时
- **则**系统返回当前阶段和进度百分比

#### 场景: 查询已完成的分析任务
- **当**前端通过 taskId 查询已完成的分析任务时
- **则**系统返回 ANALYZED 状态和分析结果摘要

### 需求: 研究空白识别
系统必须基于交叉对比分析的结论冲突、方法论缺口和数据集缺口，输出有证据支撑的研究空白。

#### 场景: 从多样化论文中识别空白
- **当**对涵盖多个子主题的论文进行分析时
- **则**系统输出分类后的研究空白，每空白附带支撑论文引用（arxiv_id）和具体证据

#### 场景: 未发现明显空白
- **当**分析的论文过少或范围过窄时
- **则**系统返回"未发现显著空白"的消息，并建议扩大搜索范围

#### 场景: 研究空白分类
- **当**输出研究空白时
- **则**每个空白必须属于以下分类之一：未探索的问题领域、矛盾结论、方法学局限、数据集不足

### 需求: Base 论文推荐
系统必须推荐 3-5 篇适合后续创新的 base 论文，每篇附带理由说明其为何适合作为起点，以及可以构建的具体创新方向。

#### 场景: 推荐 base 论文
- **当**分析成功完成时
- **则**系统返回 3-5 篇 base 论文，附带推荐理由和建议的创新方向

#### 场景: 用户选择 base 论文
- **当**用户将一篇推荐论文标记为 base 论文时
- **则**系统保存该选择并将其关联到项目

### 需求: 创新点生成
对于每篇推荐的 base 论文，系统必须生成 2-4 个具体可执行的创新点，包含预估难度（低/中/高）和预期贡献类型（方法改进、新应用、效率优化）。

#### 场景: 生成创新点
- **当**选择一篇 base 论文后
- **则**系统生成 2-4 个创新点，包含详细描述、难度评级、贡献类型、所基于的交叉分析缺口

#### 场景: 查看创新点详情
- **当**用户查看某个创新点时
- **则**系统展示描述、难度、贡献类型、所基于的 base 论文以及支撑该创新点的交叉分析证据
