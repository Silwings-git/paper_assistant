## 新增需求

### 需求: LLM 提供者抽象
系统必须为 LLM 提供者定义一个抽象接口，支持文本补全和结构化输出（JSON）解析。

#### 场景: 在提供者之间切换
- **当**系统配置了多个 LLM 提供者时
- **则**请求可以路由到任何已配置的提供者，而无需更改应用逻辑

#### 场景: 结构化输出解析
- **当**LLM 响应预期为 JSON 格式时
- **则**系统解析并验证响应是否符合预期结构

### 需求: DashScope 集成
系统必须集成阿里百炼 (DashScope) API，支持 qwen-max、qwen-plus、qwen-turbo 等模型。

#### 场景: 调用 DashScope API
- **当**请求被路由到 DashScope 时
- **则**系统使用配置的 API 密钥进行认证，并返回有效响应

#### 场景: 处理 DashScope 限流
- **当**DashScope 返回限流错误时
- **则**系统使用指数退避策略重试，最多 3 次

### 需求: 模型配置管理
系统必须在数据库中存储 LLM 提供者配置（base URL、API 密钥、模型 ID），支持启用/禁用提供者和模型。

#### 场景: 添加新的提供者
- **当**管理员添加新的 LLM 提供者配置时
- **则**提供者以加密的 API 密钥存储，并立即可用

#### 场景: 禁用模型
- **当**一个模型被标记为禁用时
- **则**该模型在前端不再可选

### 需求: 基于场景的模型路由
系统必须支持为不同使用场景（如分析、撰写）分配不同的模型，允许用户为每个任务选择性价比最优的模型。

#### 场景: 为场景分配模型
- **当**用户为一个模型分配"analysis"场景时
- **则**所有分析请求默认使用该模型

#### 场景: 回退到默认模型
- **当**没有模型被分配到某个场景时
- **则**系统使用全局配置的默认模型

### 需求: LLM 超时与重试配置
系统必须为所有 LLM 请求配置统一的超时时间（默认 120 秒）和指数退避重试策略（最多 3 次）。

#### 场景: LLM 请求超时
- **当**LLM 响应超过 120 秒时
- **则**系统抛出超时异常，记录到日志并返回给调用方

#### 场景: LLM 请求限流重试
- **当**LLM 提供者返回限流错误时
- **则**系统使用指数退避策略（1s, 2s, 4s）重试，最多 3 次

#### 场景: LLM 请求认证失败
- **当**LLM 提供者返回认证错误（如 API Key 无效）时
- **则**系统不进行重试，立即返回错误给调用方

### 需求: LlmProvider 接口契约
系统必须定义统一的 LlmProvider 接口，所有 LLM 提供者实现该接口。

#### 场景: 普通文本对话
- **当**调用 chat(systemPrompt, userPrompt) 时
- **则**返回 LLM 生成的文本响应

#### 场景: 结构化输出
- **当**调用 chatStructured(systemPrompt, userPrompt, Class<T>) 时
- **则**返回符合指定 Java 类型的结构化对象，内部使用 JSON Schema 验证

#### 场景: 提供者信息查询
- **当**调用 getProviderName() 时
- **则**返回提供者标识（如 "dashscope"）

#### 场景: 支持模型查询
- **当**调用 getSupportedModels() 时
- **则**返回该提供者支持的所有模型 ID 列表

### 需求: 启动时加载可用 Provider
系统必须在应用启动时从数据库加载所有已启用的 LLM provider 配置，并注册到工厂中。

#### 场景: 启动时加载默认 DashScope 配置
- **当**应用首次启动且数据库中存在 DashScope 配置时
- **则**系统将 DashScope provider 注册到工厂，分析请求默认路由到 qwen-plus 模型

#### 场景: 启动时无可用 Provider 配置
- **当**应用启动但数据库中没有任何 LLM provider 配置时
- **则**系统启动失败并输出明确的错误日志
